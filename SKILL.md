# PPT Creator —— 结构化演示文稿生成

## 风格资源

本技能支持多种预设风格，风格定义存储在 `references/styles/` 目录下。

### 可用风格

| 风格名称 | 文件路径 | 特点 |
|:---|:---|:---|
| **联通红风格** | `references/styles/liantong-red.md` | 3D玻璃质感、高信息密度、商务科技、浅色背景 |

### 风格使用规则

1. **默认风格**：当用户没有指定风格时，默认使用**联通红风格**
2. **切换风格**：如果用户要求更换风格，先读取对应的风格文件，然后应用
3. **修改配色**：如果用户要求修改配色（如"把红色改成蓝色"）：
   - 保留所有其他元素（3D效果、玻璃质感、图标风格、布局等）
   - 仅替换配色方案中的主色调值
   - 保持背景、文字色、图标风格不变
   - 更新提示词中的颜色值，重新生成页面

### 风格加载方式

在生成页面时，根据当前选择的风格，读取对应的风格文件，提取以下内容：
- 配色方案（主色、辅色、强调色、背景色、文字色）
- 视觉元素（背景、3D效果、图标风格、数据流、图形装饰）
- 布局规范（信息密度、对齐规则、留白）
- 文字规范（字体、禁止文字、推荐用词）
- 页面类型模板（封面页、架构图页、内容页、案例页）

## 核心工作流

采用"先讨论规划，后生成执行"的模式，确保PPT内容有深度、有逻辑、有美感。

### Phase 1: 需求理解与大纲讨论

**目标**：确定PPT要传达的核心信息，以及分成几个主要部分。

**操作步骤**：
1. 询问用户PPT的使用场景（汇报、介绍、培训、销售等）
2. 询问目标受众（领导、客户、同事、学生等）
3. 询问时间限制（决定页数多少）
4. 如果用户提供了文件夹或文档，先读取内容提取关键信息
5. 与用户讨论确定 **3-5个主要部分**（大标题）
   - 每个部分应有一个清晰的核心主题
   - 部分之间要有逻辑递进关系
   - 建议结构：引言/背景 → 核心内容（2-3部分）→ 总结/展望

**输出**：
```
## PPT大纲框架
- 主题：[PPT主题]
- 受众：[目标受众]
- 预计页数：[X页]
- 主要部分：
  1. [第一部分标题]
  2. [第二部分标题]
  3. [第三部分标题]
  ...
```

### Phase 2: 内容方向讨论

**目标**：确定每个主要部分要涵盖的大方向内容。

**操作步骤**：
1. 逐个讨论每个主要部分的内容方向
2. 每个部分讨论2-4个核心要点
3. 确保要点之间有逻辑关联（并列、递进、因果等）
4. 询问用户是否有特别想强调的内容或数据

**输出**：
```
## 内容规划
### 第一部分：[标题]
- 要点1：[内容方向]
- 要点2：[内容方向]
- 要点3：[内容方向]

### 第二部分：[标题]
...
```

### Phase 3: 页数分配与逻辑梳理

**目标**：将内容分配到具体页数，确保每页内容聚焦。

**操作步骤**：
1. 根据内容多少，为每个部分分配页数
2. 每页聚焦一个核心观点
3. 确保页与页之间有逻辑衔接
4. 规划过渡页（部分之间的衔接页）

**页数建议**：
- 封面：1页
- 目录：1页
- 每个主要部分：2-4页内容页 + 1页过渡页
- 总结/展望：1页
- 结束页：1页

**输出**：
```
## 页面规划（共X页）
P1 - 封面：[主题]
P2 - 目录
---
P3 - 过渡页：第一部分 [标题]
P4 - [第一页内容标题]
P5 - [第二页内容标题]
---
P6 - 过渡页：第二部分 [标题]
...
---
P[X] - 总结与展望
P[X+1] - 结束页
```

### Phase 4: 风格确定

**目标**：确定PPT的视觉风格。

**操作步骤**：
1. **默认使用联通红风格**：如果用户没有指定风格，直接应用联通红风格
2. 如果用户要求修改配色：
   - 询问新的主色调
   - 保留联通红风格的所有其他元素（3D效果、玻璃质感、图标风格、布局等）
   - 仅替换主色调值
   - 更新提示词中的颜色值
3. 如果用户要求完全不同的风格：
   - 询问风格偏好或参考PPT
   - 根据内容类型推荐其他风格

**联通红风格配色方案**：
- 主色调：`#E60012`（联通红）
- 辅色调：`#0066FF`（科技蓝）
- 强调色：`#FF6600`（活力橙）
- 背景色：`#F5F7FA` → `#FFFFFF`（浅灰白渐变）
- 标题文字：`#333333`
- 正文文字：`#666666`

### Phase 5: 框架页制作

**目标**：生成PPT的封面、目录、过渡页、结束页。

**操作步骤**：
1. 先制作封面页（P1）
2. 制作目录页（P2）
3. 为每个主要部分制作过渡页
4. 制作总结页和结束页

**提示词生成规则**：
- 读取当前风格文件的页面类型模板
- 将用户内容填入模板
- 确保风格一致性（颜色、图标、布局）

**执行**：
- 使用Gemini NanoBanana 2模型生成图片
- 第一次生成时向用户请求API key
- 保存生成的图片到工作目录

### Phase 6: 内容页逐页制作

**目标**：逐个部分、逐页生成PPT内容页。

**操作步骤**：
1. 从第一部分开始
2. 每页先讨论内容要点：
   - 这页的核心观点是什么？
   - 需要展示什么信息？
   - 用什么方式呈现（文字、图表、图片、流程图）？
3. 内容确定后，根据当前风格生成提示词
4. 生成页面图片
5. 确认后继续下一页

**迭代规则**：
- 每页生成后展示给用户确认
- 用户可要求调整内容或风格
- 调整后再生成新版本

### Phase 7: 整合与输出

**目标**：汇总所有页面，提供最终交付物。

**操作步骤**：
1. 列出所有生成的页面文件
2. 提供页面顺序清单
3. 询问用户是否需要调整
4. 组装成PPTX文件
5. 添加演讲者备注

## Gemini NanoBanana 2 图片生成规范

### API调用方式

**API端点**：
```
https://generativelanguage.googleapis.com/v1beta/models/gemini-3.1-flash-image-preview:generateContent?key={API_KEY}
```

**认证方式**：`x-goog-api-key` header

**请求体**：
```python
import json
import base64
import urllib.request

def generate_image(prompt, api_key, output_path):
    """
    使用Gemini NanoBanana 2生成PPT页面图片
    """
    url = "https://generativelanguage.googleapis.com/v1beta/models/gemini-3.1-flash-image-preview:generateContent?key=" + api_key
    
    body = {
        "contents": [{
            "parts": [{"text": prompt}]
        }],
        "generationConfig": {
            "responseModalities": ["IMAGE"],
            "imageConfig": {
                "imageSize": "2K",  # 或 "4K"
                "aspectRatio": "16:9"
            }
        }
    }
    
    req = urllib.request.Request(
        url,
        data=json.dumps(body).encode("utf-8"),
        headers={"Content-Type": "application/json"},
        method="POST"
    )
    
    with urllib.request.urlopen(req, timeout=300) as resp:
        result = json.loads(resp.read().decode("utf-8"))
    
    # 提取图片数据
    for candidate in result.get("candidates", []):
        for part in candidate.get("content", {}).get("parts", []):
            inline_data = part.get("inlineData")
            if inline_data and inline_data.get("data"):
                image_bytes = base64.b64decode(inline_data["data"])
                with open(output_path, "wb") as f:
                    f.write(image_bytes)
                return output_path
    
    raise RuntimeError("No image returned")
```

### 重试机制

**必须实现重试机制**：
```python
def generate_with_retry(prompt, api_key, max_attempts=8):
    """带重试机制的图片生成"""
    for attempt in range(1, max_attempts + 1):
        try:
            return generate_image(prompt, api_key, output_path)
        except urllib.error.HTTPError as e:
            if e.code == 429:  # Rate limit
                wait_time = min(2 ** attempt, 60)  # 指数退避
                time.sleep(wait_time)
            else:
                time.sleep(2)
        except Exception as e:
            print(f"Attempt {attempt} failed: {e}")
            time.sleep(2)
    
    raise RuntimeError(f"Failed after {max_attempts} attempts")
```

**重试策略**：
- 最多8次重试
- HTTP 429错误：指数退避（2^attempt秒，最多60秒）
- 其他错误：固定2秒间隔
- 超时设置：300秒

### 绝对禁止项（CRITICAL CONSTRAINTS）

**以下约束必须严格遵守，每次生成提示词时必须包含：**

1. **禁止颜色代码和颜色名称作为画面文字**：
   - ❌ 绝对禁止将任何颜色十六进制代码（如 `#E60012`、`#0066FF`、`#FF6600`）作为文字写在画面上
   - ❌ 绝对禁止将颜色名称（如"联通红"、"科技蓝"、"活力橙"、"深灰"、"浅灰"）作为文字写在画面上
   - ✅ 颜色只用于描述视觉效果（如"使用红色作为主色调"），绝不能出现在画面文字内容中

2. **禁止占位符和虚假联系信息**：
   - ❌ 绝对禁止生成 `[具体步骤]`、`[你的电话]`、`[公司名称]`、`[汇报人]`、`[日期]`、`[联系方式]` 等占位符
   - ❌ 绝对禁止生成 "请联系销售"、"拨打热线"、"访问官网" 等引导性占位文字
   - ✅ 直接删除这些元素，页面中不要出现任何需要填写的位置

3. **禁止乱码和捏造词汇**：
   - ❌ 绝对禁止出现无法识别的乱码汉字（如"銆"、"堟"、"犮"等）
   - ❌ 绝对禁止捏造与大纲无关的词汇或概念
   - ✅ 使用标准清晰的黑体/微软雅黑字体，确保所有文字可识别、可理解

4. **风格一致性强制要求**：
   - 每一页的提示词都必须包含完整的风格要求描述
   - 禁止在生成过程中逐渐偏离初始风格
   - 禁止后半部分页面出现与前半部分不同的视觉风格（如突然变成深色背景、突然使用不同图标风格等）
   - 所有页面必须使用统一的配色、字体、图标风格、布局规范

## PPT组装规范

### 组装流程

使用python-pptx库组装PPT：

```python
from pptx import Presentation
from pptx.util import Inches

def build_ppt(image_dir, output_path, notes_dict=None):
    """
    将图片组装成PPT，并添加演讲者备注
    
    Args:
        image_dir: 图片文件夹路径
        output_path: 输出PPTX文件路径
        notes_dict: 演讲者备注字典 {filename: notes_text}
    """
    prs = Presentation()
    prs.slide_width = Inches(13.333)  # 16:9比例
    prs.slide_height = Inches(7.5)
    blank_layout = prs.slide_layouts[6]  # 空白布局
    
    # 按页码排序
    image_files = sorted([f for f in os.listdir(image_dir) if f.endswith('.png')])
    
    for image_file in image_files:
        slide = prs.slides.add_slide(blank_layout)
        slide.shapes.add_picture(
            os.path.join(image_dir, image_file),
            0, 0,
            width=prs.slide_width,
            height=prs.slide_height
        )
        
        # 添加演讲者备注
        if notes_dict and image_file in notes_dict:
            notes_slide = slide.notes_slide
            notes_text_frame = notes_slide.notes_text_frame
            notes_text_frame.text = notes_dict[image_file]
    
    prs.save(output_path)
```

### 图片命名规范

- 按页码顺序命名：`P01_封面页.png`, `P02_目录页.png`, ...
- 使用两位数页码，确保排序正确
- 文件名包含页面内容简述

### 演讲者备注规范

**必须添加演讲者备注**，备注内容要求：

1. **信息密度有起伏**：
   - 重点页（架构图、核心技术）：详细展开，5-6分钟
   - 过渡页（目录、总结）：快速带过，1-2分钟
   - 普通内容页：清晰阐述，3-4分钟

2. **时长标注**：每页备注开头标注建议时长
   ```
   【重点页】5-6分钟
   这里是详细讲解内容...
   ```

3. **互动提示**：关键页添加互动建议
   ```
   【互动】各位，你们的知识库是不是也面临这些问题？
   ```

4. **关键转折**：标注重要的转折点和强调内容
   ```
   【关键转折】AI就绪度——你的知识库准备好被AI消费了吗？
   ```

5. **避免过度承诺**：使用软性描述，不用具体数字
   - ✅ "效率显著提升"
   - ❌ "效率提升40%"

## 风格一致性控制

### 必须遵守的规则

1. **颜色统一**：
   - 所有图标必须使用当前风格的配色方案
   - 禁止出现彩色图标（如Copilot的彩虹色）
   - 主色、辅色、强调色必须全局一致

2. **Logo处理**：
   - 明确哪些页面需要Logo，哪些不需要
   - 需要Logo的页面必须同时显示Logo图标和公司名称
   - 通常仅封面和封底需要Logo

3. **文字规范**：
   - 禁止出现"内部参考"、"仅供内部交流"等占位文字
   - 禁止出现"汇报人"、"汇报日期"等需要手工填写的项目
   - 标题避免主观词汇（"观点"、"总结与呼吁"），改用客观词汇（"回顾"、"下一步"）

4. **布局规范**：
   - 同类元素必须水平对齐
   - 避免头重脚轻，上下布局均衡
   - 确保所有文字完整显示，不被截断

5. **强调规范**：
   - 不要随意标红强调
   - 标红必须有明确的技术含义或业务含义
   - 如果标红需要解释，确保演讲者备注中有说明

## 内容设计原则

### 该做的（DO）

1. **先讨论规划，后生成执行**
2. **确保内容有逻辑性**：页与页之间有逻辑衔接
3. **重点突出**：核心内容用更多页面展开
4. **技术深入**：方法论部分讲清底层实现逻辑
5. **案例支撑**：用实际案例验证方案可行性
6. **渐进演进**：强调从传统到AI的平滑过渡
7. **用户确认**：每个主要阶段完成后都要用户确认
8. **保存进度**：定期总结当前进度

### 不该做的（DON'T）

1. **不要假设库可用**：使用python-pptx前确认已安装
2. **不要出现品牌字样**：参考内容需改写，去除品牌标识
3. **不要案例指名道姓**：使用"某大型集团"、"某银行"等匿名
4. **不要过度承诺**：避免具体百分比数字，用软性描述
5. **不要观点性词汇**：不用"观点"、"总结与呼吁"等主观词汇
6. **不要彩色图标**：所有图标必须统一配色
7. **不要占位文字**：禁止"内部参考"、"汇报人"等
8. **不要随意标红**：标红必须有明确含义
9. **不要信息平均**：重点内容重点讲，非重点简略讲
10. **不要深色背景**：除非用户明确要求，默认使用浅色背景

## 迭代优化流程

### 分批次生成

1. **先生成示例页**：生成1-2个示例页面确认风格
2. **用户确认风格**：确认无误后再批量生成
3. **分部分生成**：按大纲部分分批生成，避免一次性生成过多

### 版本管理

1. **统一文件夹**：最终版本统一放在一个文件夹
2. **保留历史**：修改前的版本保留在单独文件夹
3. **命名规范**：修改后的页面标注"修改版"或"v2"

### 修改响应

当用户提出修改意见时：
1. 明确修改范围：哪些页面需要修改，哪些保留
2. 重新生成修改页面
3. 统一整合所有页面到新文件夹
4. 确认无误后组装PPT

## 对话模式参考

参考Office Hour的讨论方式：
- **引导式提问**：通过问题引导用户思考，而非直接给答案
- **结构化输出**：每个阶段都有明确的输出格式
- **确认机制**：关键决策点都要用户确认
- **迭代优化**：允许用户随时调整已确定的内容
